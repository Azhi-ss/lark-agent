# 飞书周报 Agent — 前端设计方案 (Design Brief for Stitch)

> 用途:把当前已实现的前端方案提炼为设计规格,供在 Google Stitch 上探索更优的可视化方案。
> 现状技术栈:Vue 3 + Vite + Tailwind CSS v4。下文为产品/设计层描述,与框架无关。

---

## 1. 产品定位

一个可视化 Web 应用:用户粘贴飞书周报链接 → AI agent 提炼/修改周报 → 界面逐处展示 diff → 确认后一键写回飞书。

核心价值:**让 AI 编辑文档是"可审阅、可控、可回滚"的**,而非黑盒直接改写。

---

## 2. 信息架构(IA)

整页为 **三段式纵向布局**:顶部 Header(全局) + 主体左右双栏(1:1)。

```
┌─────────────────────────────────────────────────────────────┐
│  Header: 标题 | 模型标识 | [飞书URL输入框........] [加载文档]  │
├──────────────────────────┬──────────────────────────────────┤
│                          │  指令区(输入框 + 处理按钮)        │
│                          ├──────────────────────────────────┤
│   文档渲染区(只读)        │  Agent 输出区                    │
│   - 文档元信息            │   - 思考过程(折叠)               │
│   - Markdown 正文         │   - 文本回复                     │
│                          │   - 修改 diff 列表(逐处)         │
│                          ├──────────────────────────────────┤
│                          │  操作区:确认并写回飞书(底部固定) │
└──────────────────────────┴──────────────────────────────────┘
```

## 3. 组件清单

| 区域 | 组件 | 职责 | 关键状态 |
|---|---|---|---|
| Header | DocUrlBar | 飞书文档 URL 输入 + 加载按钮 | loading / idle |
| Header | BrandTag | 产品名 + 模型标识 | — |
| 左栏 | DocMeta | document_id / revision / 块数 | — |
| 左栏 | MarkdownViewer | 渲染周报正文(h1-3, ul, ol, code, pre, strong) | empty / loaded |
| 右栏 | InstructionBar | 多行指令输入 + 预填默认指令 + 处理按钮 | running / idle |
| 右栏 | ThinkingCard | agent 思考过程,灰底小字 | 折叠/展开 |
| 右栏 | AgentReply | agent 文本回复 | — |
| 右栏 | DiffCard × N | 每处修改卡片:序号 + 原因 + 原文(红) + 替换为(绿) | — |
| 右栏底 | ApplyBar | 写回飞书按钮 + 数量提示 + 安全文案 | applying / done / partial |
| 全局 | ErrorBanner | 加载/写回错误红条 | — |

---

## 4. 交互流程(用户旅程)

1. **加载** → 输入 URL,点"加载文档",左栏渲染正文 + 元信息
2. **下达指令** → 右栏指令框预填"提炼本周要点…",用户可改,点"让 Agent 处理"
3. **流式反馈** → 右栏依次出现:思考过程(灰)→ 文本回复 → diff 卡片(逐处)
4. **审阅 diff** → 每处修改独立卡片,红=原文匹配,绿=替换为,附修改原因
5. **写回** → 检查无误后点底部"确认并写回飞书(N 处修改)",显示成功/部分失败

**安全设计**:写回按钮文案明示"将直接修改飞书文档";支持 `LARK_WRITEBACK=0` 全局模拟模式。

---

## 5. 数据流(API 契约)

| 动作 | 方法 | 端点 | 输入 | 输出 |
|---|---|---|---|---|
| 加载文档 | POST | `/api/doc/load` | `{url}` | `{markdown, document_id, revision_id, block_count}` |
| Agent 处理 | POST | `/api/agent/chat` | `{markdown, instruction}` | SSE 流:`thinking` / `text` / `done{replacements[]}` / `error` |
| 写回飞书 | POST | `/api/doc/apply` | `{url, replacements[], revision_id:-1}` | `{ok, count, results[]}` |

`replacements[i] = {pattern: 原文匹配, content: 替换为, reason: 修改原因}`

## 6. 视觉风格(Design Tokens)

| Token | 当前值 | 说明 |
|---|---|---|
| 主色 | `violet-600` (#7c3aed) | 加载/处理按钮、diff 卡片头、聚焦边框 |
| 主色 hover | `violet-700` | — |
| 成功色 | `green-600` (#16a34a) | 写回按钮 |
| 危险色 | `red-50/600/800` | 原文区、错误条 |
| 新增色 | `green-50/800` | 替换为区 |
| 思考色 | `gray-100/400` | 思考卡片背景/文字 |
| 背景 | `#f7f7f8` (body) / `white` (卡片) | — |
| 字体 | system-ui, 'Segoe UI', Roboto | 中文优先系统字体 |
| 圆角 | `rounded` (4px) 组件 / `6px` 代码块 | — |
| 正文行高 | 1.6 | markdown 正文 |

**当前风格定性**:轻量、克制、偏工具型。紫色主色 + 白底卡片 + 灰底容器,信息密度高但视觉较平。

---

## 7. 当前方案的不足(Stitch 可优化方向)

1. **diff 可读性**:纯文本红绿块,长文本对比吃力 → 可改用逐行/逐词高亮 diff
2. **写回风险感知弱**:一个绿色按钮不够"危险提示" → 可加二次确认、影响范围预览、版本对比
3. **左栏文档只读无导航**:长周报难定位 → 可加目录锚点、agent 修改处高亮跳转
4. **指令交互单薄**:只有一次文本指令 → 可支持多轮对话、指令历史、快捷指令模板
5. **空状态/加载态粗糙**:加载中无骨架屏,空态仅一行灰字
6. **移动端未适配**:1:1 双栏在小屏不可用
7. **缺写回历史/撤销**:写回后无法回退 → 可加版本时间线

---

## 8. Stitch 输入建议

把第 2-6 节作为结构输入,第 7 节作为优化目标。建议向 Stitch 提的探索方向:

- 方向 A:更专业的"文档审阅"体验(类 GitHub PR review:行内批注 + 侧边讨论)
- 方向 B:更对话化的"AI 副驾"体验(agent 为主,文档为辅,流式气泡)
- 方向 C:更安全的"版本管理"体验(写回前预览 + 时间线 + 一键回滚)
